# CNC 培訓班數位化策略：最終方案

> 基於 123 步驟全流程、30 個工具、4 角色旅程、PRD/TRD/MRD/BRD 的完整分析

**版本**：v2.0（取代 v1.0 三階段混合策略）
**日期**：2026-02-19

---

## 結論：直接建平台，三階段上線

**不做 Phase 0 流程簡化+Serverless 過渡。直接做自建平台。**

上一版建議「D+C+A 三階段混合」，先花 NT$50-100K 做 Apps Script / Cloud Functions 自動化，再建平台。這個建議是錯的，原因如下：

1. **Phase 0 的 Serverless 自動化（Apps Script、Cloud Functions）在 Phase 1 平台上線後就要丟掉**。報名成功信、金流 webhook、影片下架排程——這些功能平台本身就會做。Phase 0 花的 NT$50-100K 和 4 週時間，不會沉澱為任何長期資產。

2. **流程簡化不需要獨立 Phase**。建平台的需求分析階段本來就會做流程梳理。123 步驟哪些要自動化、哪些要砍掉、哪些保留人工，這是 PRD 的一部分，不是一個獨立的前置專案。

3. **Phase 0 給出「漸進式驗證」的安全感是假的**。真正需要驗證的假設（學員接受新系統、團隊維護能力、督導配合度）只有在平台上線後才能驗證。Apps Script 跑得動不代表平台跑得動，兩者驗證的是完全不同的東西。

所以，正確的做法是：**砍掉 Phase 0，直接從平台 MVP 開始，分三階段上線。**

## 一、為什麼現在就該建平台

### 1.1 規模已經到了臨界點

| 指標 | 數據 | 含義 |
|------|------|------|
| 年學員數 | 135 人（2025），趨勢 CAGR 33% | 2027 年將突破 250 人 |
| 年梯次 | 4 梯 | 每梯之間助教幾乎沒有喘息 |
| 每梯次行政工時 | 60-90 小時 | 4 梯 = 240-360 小時/年 = 約 1.5 個全職人力的行政時間 |
| 工具數量 | 30 個 | 資料散落 5+ 個地方，無 Single Source of Truth |
| 年營收 | ~NT$12M（根據 BRD 估算） | 有能力支撐 NT$1.5-2M 的系統投資 |

**關鍵判斷**：CNC 不是一個「不確定要不要數位化」的階段。MRD 已經定了 3 年目標是 300 人/年、8-10 梯。BRD 已經做了 ROI 分析，結論是 6 個月回本。PRD 和 TRD 已經把技術架構設計到 API 端點的程度。這不是「要不要做」的問題，是「怎麼做最快上線」的問題。

### 1.2 低成本方案無法解決核心問題

上一版分析了 4 種方案（全平台、API 串連、Serverless、流程簡化），結論是對的：**方案 B/C/D 都無法作為終態**。

- **流程簡化（D）**：只能省 15-25% 行政時間。123 步驟裡真正能「砍掉」的不多（牌卡演練會合併到見面會？這種省 3 小時的事不值得當策略）
- **Serverless（C）**：Zapier/n8n 超過 50 個流程後就是義大利麵。CNC 光報名到認證就有 7 個大階段，每階段 10-20 個自動化點，很快就超標
- **API 串連（B）**：glue code 脆弱，Google API 變更就斷。且學員體驗沒有改善——仍然要在 5 個平台之間跳

**既然 B/C/D 都只是過渡，為什麼要花錢和時間做過渡？直接做終態。**

### 1.3 「先驗證再投入」的邏輯陷阱

上一版的核心論述是「先低成本驗證，再決定是否投入平台」。聽起來合理，但放到 CNC 的具體情境不成立：

- **假設 A-01（團隊維護能力）**：CNC 目前沒有工程師。無論做 Phase 0 的 Cloud Functions 還是做 Phase 1 的平台，都需要技術人力。Phase 0 無法驗證「團隊是否能維護平台」，因為 Cloud Functions 和 Web 平台是完全不同的維護內容。
- **假設 A-02（學員接受遷移）**：學員目前用 Google Forms 報名、Gmail 收信。從 Google Forms 換到「自建報名頁」的遷移成本接近零——學員根本不在乎報名表長什麼樣，他們在乎的是報名後能不能立刻收到確認。
- **假設 A-08（學長姐/審核導師配合）**：這個假設在 Phase 2（督導+認證）才會觸碰。Phase 0 的 Apps Script 完全不涉及。

**結論：Phase 0 驗證不了任何對平台有意義的假設。它只是在拖時間。**

## 二、三階段方案

### Phase 1：報名+金流+通知 MVP（8 週 | NT$600-800K）

這一階段解決最大的行政痛點（報名追帳、手動寄信），同時建立 Single Source of Truth。

| 模組 | 功能 | 替代的現有工具 | 省時 |
|------|------|--------------|------|
| 線上報名 | 梯次選擇、報名上限、截止自動關閉、報名表前端驗證 | Google Forms | - |
| 金流整合 | 藍新 MPG（信用卡、ATM 虛擬帳號）、手動匯款確認、電子發票 | 手動追帳 + 人工開發票 | **6-8 hr/梯** |
| 自動通知 | 16 種 Email 範本（報名確認、課前通知 1&2、認證提醒等）、排程發送、發送紀錄 | Gmail BCC | **8-12 hr/梯** |
| 管理後台 | 報名儀表板、付款狀態、學員名冊、批次操作、Excel 匯出 | Google Sheets | **4-6 hr/梯** |
| OAuth 登入 | Google 帳號一鍵登入 + Email/Password 備案 | 無（新增） | - |
| 換證追蹤 | 自動計算到期日、排程寄送提醒信（6/3/1 個月前） | 人工 Excel 追蹤 | **2-4 hr/次** |

**Phase 1 不做的事**：
- 課程管理（繼續用 Google Classroom）
- 督導預約（繼續用 Google Calendar + Calendly）
- 認證審核流程（繼續用 Email + Drive）
- 作業繳交（繼續用 Google Drive）
- 學員個人中心（Phase 2）

**Phase 1 預期效果**：
- 行政助教減負：**-40% ~ -50%**（從 60-90hr 降至 30-50hr/梯次）
- 學員體驗：報名→付款→收到確認信，全程 5 分鐘自動完成
- 建立 Single Source of Truth：所有學員資料從報名系統出發

**Phase 1 啟動前置作業（第 0 週，與開發並行）**：
1. 藍新商家帳號申請送件（審核需 1-2 週）
2. SendGrid 帳號設定
3. GCP 環境建立（Cloud Run + Cloud SQL + Cloud Storage）
4. 與行政助教開 1-2 次會議，確認流程簡化項目（哪些通知可合併、哪些步驟可砍）

**注意**：流程簡化在這裡做，不需要獨立 Phase。這就是 PRD 需求分析的一部分。

---

### Phase 2：課程管理+督導+認證+學員中心（8 週 | NT$600-700K）

**啟動條件**：Phase 1 穩定運行至少 1 個完整梯次。

| 模組 | 功能 | 替代的現有工具 | 省時 |
|------|------|--------------|------|
| 梯次建立精靈 | 一鍵複製上梯設定、5 分鐘建新梯次 | 手動建 Classroom + Drive + Meet | **8-10 hr/梯** |
| 學員中心 | 統一入口：課程日曆、公告、作業區、進度追蹤 | Classroom + LINE + Email + Drive | 學員不用切 5 個平台 |
| 作業管理 | 附件 A-E 線上繳交、自動時數加總（>=24hr 檢核）、進度矩陣 | Google Drive + 人工逐一查 | **4-6 hr/梯** |
| 督導預約 | 統一預約入口、公平開放時間、防超額、自動行事曆同步 | Google Calendar + Calendly（3 督導各用不同工具） | **4-6 hr/梯** |
| 認證工作流 | 資格自動檢核 → 筆試發送 → 審核導師分配 → 進度追蹤 → 結果通知 | Email + Drive + 人工逐一操作 | **6-8 hr/梯** |
| 證書管理 | PDF 自動套版（姓名/日期/編號）、數位下載、列印版匯出 | PowerPoint 套版 + 影印店 | **3-4 hr/梯** |
| 學長姐媒合 | AI 推薦配對（產業/職位相關性）、管理員確認、一鍵通知 | Google Sheets + AI 手動操作 + 多管道公告 | **5-8 hr/梯** |

**Phase 2 預期效果**：
- 行政助教減負：**-60% ~ -70%**（從 60-90hr 降至 20-30hr/梯次）
- 學員體驗：一站式入口，進度透明，督導預約公平
- 可擴展性：支撐 6-8 梯/年、250+ 人/年

---

### Phase 3：補課影片+開課快手替代+儀表板+CRM+校友管理（6 週 | NT$300-400K）

**啟動條件**：Phase 2 穩定運作。

| 模組 | 功能 | 替代的現有工具 | 省時 |
|------|------|--------------|------|
| 補課影片管理 | YouTube 不公開連結嵌入、觀看期限自動控制、到期自動關閉 | 手動上傳 YouTube + 手動下架 x13 堂 | **13-20 hr/梯** |
| 預錄課程管理 | 開課快手替代、學員自動開通、觀看期限控制 | 開課快手手動匯入 | **2-4 hr/梯** |
| **營運儀表板** | 報名數、付款率、認證通過率、換證到期人數 | Excel 手動統計 | **2-3 hr/次** |
| **學員 CRM 列表** | 搜尋/篩選、狀態標籤（潛在→報名→上課→認證→校友）、匯出 | Google Sheets | **1-2 hr/次** |
| **校友管理** | 認證師名單、證書狀態、換證歷史 | Excel + 人工追蹤 | **2-4 hr/次** |

**Phase 3 預期效果**：
- 行政助教減負：**累計 -70% ~ -83%**（從 60-90hr 降至 10-20hr/梯次）
- 數據決策：即時掌握營運關鍵指標，不需手動彙整 Excel
- 內容管理：補課影片與預錄課程全自動化，完全替代開課快手

## 三、成本與 ROI

### 3.1 總投資

| 項目 | Phase 1 | Phase 2 | Phase 3 | 合計 |
|------|---------|---------|---------|------|
| 人力成本 | NT$500-650K | NT$500-600K | NT$250-400K | NT$1.25-1.65M |
| 基礎設施（年） | NT$120K | 同左（已含） | 同左（已含） | NT$120K |
| 第三方服務（年） | NT$360K | 同左（已含） | 同左（已含） | NT$360K |
| 其他（法律、培訓、預備金） | NT$150K | NT$100K | NT$50K | NT$300K |
| **小計** | **NT$1.13-1.28M** | **NT$600-700K** | **NT$300-400K** | **NT$2.0-2.4M** |

這與 BRD 的預算範圍 NT$2.07M 一致。

### 3.2 ROI 計算

**成本節省**：
- 行政工時：每梯次省 50-70 小時 x 4 梯/年 = 200-280 小時/年
- 以 NT$400/小時計算 = NT$80K-112K/年
- 加上品質提升（零遺漏通知、零追帳錯誤）的隱性價值

**營收增長**：
- 系統支撐從 4 梯擴展到 6-8 梯（不加人力）
- 每梯平均營收 ~NT$3M（135 人/4 梯 = 34 人/梯，平均客單 ~NT$48K）
- 多 2-4 梯 = NT$6M-12M 增量營收
- 扣除 30% 成本率 = NT$4.2M-8.4M 淨利增量

**回收期**：即使只看成本節省，18 個月內回本。加上營收增長，BRD 的估算是 6 個月。

### 3.3 與上一版的成本比較

| 方案 | 總投資 | 行政減負 | 學員體驗 | 備註 |
|------|--------|---------|---------|------|
| v1.0 三階段（D+C+A） | NT$1.05-1.6M | -70%~-83% | 大幅改善 | Phase 0 的 NT$100K 是浪費 |
| **v2.0 兩階段（A 直上）** | **NT$1.93-2.33M** | **-70%~-83%** | **大幅改善** | 多出的成本來自不做 Phase 0 過渡、直接投入正式架構 |

表面上看，v2.0 比 v1.0 貴了 NT$0.7-0.9M。但：

1. **v1.0 的 NT$50-100K Phase 0 是沉沒成本**（平台上線後全部丟掉）
2. **v1.0 多了 4 週等待時間**，影響的是整個上線時程（晚 1 個月意味著晚 1 個梯次開始用新系統）
3. **v2.0 的「多出的成本」其實是 v1.0 少算了**——v1.0 的 Phase 1 和 Phase 2 成本估算（NT$400-600K + NT$600-900K = NT$1.0-1.5M）加上 Phase 0（NT$50-100K）= NT$1.05-1.6M，這個數字偏低。BRD 的詳細估算（含人力、基礎設施、第三方、法律、預備金）是 NT$2.07M。v2.0 的 NT$1.93-2.33M 更接近實際成本。

**結論：v1.0 不是「更便宜」，是「少算了成本」。**

## 四、時程

```
Week  0         8              16             22
  |-----------|---------------|-------------|
  Phase 1      Phase 2         Phase 3
  報名+金流     課程+督導+認證   影片+儀表板
  +通知 MVP     +學員中心       +CRM+校友

  NT$1.1-1.3M  NT$0.6-0.7M    NT$0.3-0.4M

  目標：        目標：          目標：
  2026 Q2      2026 Q3        2026 Q4
  配合下梯報名   配合新梯啟用     全面數據驅動

  驗證：        驗證：          驗證：
  - 學員接受度   - 規模化能力    - 儀表板可用性
  - 金流穩定     - Classroom替代 - CRM 實用性
  - 團隊維護     - 督導配合度    - 影片管理效率
```

**與 PRD/BRD 時程對齊**：
- PRD Phase 1（報名與金流 MVP）= 本方案 Phase 1，8 週
- PRD Phase 2（課程管理+督導+認證）= 本方案 Phase 2，8 週
- PRD Phase 3（補課影片+開課快手替代+儀表板+CRM+校友）= 本方案 Phase 3，6 週
- PRD 進階功能（Email 行銷、API 對接、進階報表）= 本方案暫緩，留到 Phase 3 穩定後再評估

**為什麼拆分為 Phase 2 + Phase 3？**

原本 14 週的 Phase 2 範圍過大，拆分為兩個更聚焦的階段：
- **Phase 2（8 週）聚焦核心教學流程**：
  - 課程管理、督導預約、認證工作流、證書管理、學長姐媒合
  - 這些功能互相依賴，必須一起上線

- **Phase 3（6 週）聚焦數據與內容管理**：
  - 補課影片管理（自動上下架）
  - 開課快手替代（預錄課程管理）
  - 營運儀表板、學員 CRM 列表、校友管理
  - **ROI 明確**：節省每月 5-10 小時 Excel 手動彙整時間，支援數據驅動決策

- **進階功能仍然暫緩**：
  - Email 行銷自動化（SendGrid 直接操作足夠）
  - ChinaNCDA API 對接（現階段無迫切需求）
  - 進階報表（同期群分析、LTV）

Phase 3 做完，行政減負已達 70-83%，學員體驗已大幅提升，且具備基本數據決策能力。

## 五、不做的事（明確排除）

| 項目 | 為什麼不做 |
|------|---------|
| Phase 0 Serverless 過渡 | 沉沒成本，建平台後全部丟掉 |
| LINE Messaging API 整合 | LINE 群組保留做社群互動，正式通知走 Email。費用高、價值低 |
| 包裹寄送自動化 | 每年只 4 次、純實體、ROI 極低 |
| 開課快手 API 整合 | 無官方 API，Phase 3 預錄課程管理模組上線後逐步替代 |
| Email 行銷自動化 | SendGrid 直接操作足夠，現階段不需自動化行銷活動排程 |
| ChinaNCDA API 對接 | 現階段無迫切需求，人工同步足夠 |
| 進階報表（同期群分析、LTV） | 基礎儀表板已足夠，進階分析等規模 >300 人/年再做 |
| 數位卡牌工具 | 差異化方向對，但現階段規模不支撐開發成本 |
| 校友接案平台 | MRD P3 層級，2027 年再評估 |
| Mobile App | 響應式 Web 足夠，不需原生 App |

## 六、風險與 Mitigation

| 風險 | 影響 | 機率 | Mitigation |
|------|------|------|-----------|
| **藍新審核不通過** | Phase 1 金流無法上線 | 低 | Week 0 立刻送件。備案：先上線手動匯款+管理後台確認，金流通過後補上線上付款 |
| **開發人力不足/延遲** | 時程延後 | 中 | Phase 1 嚴守 MVP 範圍（只做報名+金流+通知+後台）。有進度壓力時先砍 Phase 2 範圍，不砍 Phase 1 |
| **學員不習慣新系統** | 報名轉換率下降 | 低 | Phase 1 只改報名頁+Email，學員端改變極小。Phase 2 新梯用新系統、舊梯不動 |
| **行政助教抗拒** | 系統無人用 | 中 | 從最痛的點切入（追帳是所有助教最恨的事）。讓助教參與 UAT，確保系統比 Sheets 更好用 |
| **督導不配合新預約系統** | Phase 2 預約模組閒置 | 中 | 預約系統同步至 Google Calendar（督導看到的還是 Calendar），提供比 Calendly 更好的體驗 |
| **系統維護人力斷層** | 上線後無人修 Bug | 高 | GCP 全托管降低維運。建立 runbook。評估外包維護（每月固定費用比養人便宜） |
| **Phase 2/3 範圍膨脹** | 超時超預算 | 中 | Phase 2 聚焦課程+督導+認證，Phase 3 聚焦影片+CRM，各自可獨立上線。Phase 3 內容管理模組可延後 |

## 七、Phase 1 啟動前須確認事項

在正式啟動開發前，以下 5 項必須確認：

| 項目 | 說明 | 負責人 | 時限 |
|------|------|--------|------|
| **藍新商家帳號** | 送出申請，確認教育培訓業是否有額外審核要求 | 營運 | 第 1 週 |
| **開發團隊** | 確認前端 1 人 + 後端 1 人 可全職投入至少 4 個月 | 專案發起人 | 啟動前 |
| **Email 範本** | 與行政助教確認 16 種 Email 的內容、發送時機、變數 | 產品經理+助教 | 第 1-2 週 |
| **流程簡化決策** | 與助教開 1 次會議，確認可合併/砍掉的步驟（這不是獨立 Phase，是需求分析） | 產品經理+助教 | 第 1 週 |
| **預算核准** | 確認首期 NT$1.1-1.3M 預算到位 | 專案發起人 | 啟動前 |

---

## 八、核心決策摘要

1. **直接建平台，不做 Serverless 過渡** -- Phase 0 的 Cloud Functions / Apps Script 在平台上線後就是垃圾代碼。省下那 4 週和 NT$100K，直接用在平台上。

2. **三階段，不是四階段** -- Phase 1（報名+金流+通知，8 週）→ Phase 2（課程+督導+認證+學員中心，8 週）→ Phase 3（影片+儀表板+CRM+校友，6 週）。進階 CRM+數據分析暫緩，等規模到 300+ 人再做。

3. **Phase 1 = Single Source of Truth** -- 所有學員資料從報名系統出發。Google Classroom、LINE、開課快手繼續用，但資料源頭是平台。

4. **每個 Phase 的啟動條件是「前一個 Phase 穩定運作」** -- 不是時間到了就開始。如果前一個 Phase 有重大問題，先修好再推進。

5. **LINE 群組不整合，包裹不自動化，進階 CRM 不急** -- 這三件事的 ROI 不支持現階段投入。

6. **流程簡化是需求分析的一部分，不是獨立 Phase** -- 開 1-2 次會議就能決定。不需要 4 週的「Phase 0」。
